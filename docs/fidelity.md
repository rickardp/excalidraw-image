# Fidelity reference

What `excalidraw-image` reproduces from the web app, what it deliberately
does not, and how each claim is tested. Written against the v1
implementation. For architectural context, see [PLAN.md](../PLAN.md);
for the locked Phase-0 decisions, [PHASE0.md](../PHASE0.md).

## Goal

The SVG and PNG output, when rendered in a browser, should be visually
equivalent to what excalidraw.com produces, modulo the documented
divergences below.

"Equivalent" specifically means:

- Geometry, color, and stroke styles match byte-for-byte at the SVG layer
  (the same `exportToSvg` runs on both sides).
- Text metrics, line breaks, and font selection match within tight
  measured tolerances.
- Editable `.excalidraw.svg` files round-trip: re-opening on
  excalidraw.com restores the original scene.

The CLI is not a pixel-exact reproduction of any specific browser engine.
Where measurable divergences exist, they are listed below with the
mechanism and the test that locks the current behaviour in place.

## What is exact

The export path is Excalidraw's own. The JS bundle ships
`@excalidraw/excalidraw`'s `exportToSvg`, the same function the web app
calls. All of the following are produced verbatim by upstream code:

- **Element placement, sizing, colors, opacity** — direct outputs of
  upstream's renderer.
- **Stroke styles, fill patterns** — generated by the bundled rough.js
  with the same seed semantics as the web app.
- **Text rendering algorithm** — Excalidraw's own line-break and metrics
  pipeline, with our `FontkitTextMetricsProvider` replacing the browser's
  `CanvasRenderingContext2D.measureText`.
- **Font metric divergence ≤ 0.0005 px vs Chromium** for all eight
  supported families (Excalifont, Virgil, Nunito, Lilita One, Comic
  Shanns, Cascadia, Liberation Sans, Xiaolai). Validated by
  [FNT-005](../TASKS.md) against a Playwright + Chromium
  `canvas.measureText` baseline (`tests/fixtures/browser-font-baseline.json`).
- **Editable `.excalidraw.svg` round-trips** — element IDs, `files`
  payloads, and font-family metadata are preserved through export →
  decode. Locked by [FNT-011](../TASKS.md) and [E-003](../TASKS.md);
  decoder is the pako-inflate path matching upstream `data/encode.ts`.
- **Byte-identical Deno↔Rust output** — the dev-loop host (Deno) and the
  production host (`deno_core` in Rust) produce the same bytes for every
  fixture in `tests/fixtures/`. Locked by [R-007](../TASKS.md). Drift is
  treated as a host leak and a hard CI fail, not as a tolerance to widen.

## Documented divergences

Each is listed with a one-line cause, the user-visible effect, and a
pointer at the relevant code or task.

### Emoji

Segoe UI Emoji is not bundled. Excalidraw's font fallback chain keeps
`Segoe UI Emoji` in the `font-family` attribute on emoji-bearing
`<text>` elements, but no `@font-face` declaration is emitted for it.
Consequence: emoji renders correctly only if the viewer's system has the
font installed; otherwise the glyphs fall back to a system default or
appear as tofu. PNG rasterization through `resvg` has the same
limitation — `fontdb` is loaded only with the bundled WOFF2 set.

See [PLAN.md §4A.6](../PLAN.md) (the `local:` descriptor trap),
[FNT-010](../TASKS.md).

### Helvetica

Helvetica is not bundled. Text marked as Helvetica is measured against
the bundled Liberation Sans (which has near-equivalent metrics), but the
emitted SVG keeps `font-family="Helvetica, sans-serif, Segoe UI Emoji"`
verbatim so editable round-trips preserve the user's intent. In a
browser with Helvetica installed, the SVG will render in Helvetica; in
one without it, the fallback chain takes over.

See [PLAN.md §4A.5](../PLAN.md), [FNT-008](../TASKS.md).

### Embeddables (iframes, YouTube, video)

Not rendered. `renderEmbeddables: false` is hard-coded in the JS export
options. Scenes that include embeddable elements export the underlying
geometry (the placeholder rectangle) but not the embedded content.

See [PLAN.md §1](../PLAN.md) — non-goals for v1.

### OS-local and user-installed fonts

Not supported. Only the eight families bundled as WOFF2 from
`@excalidraw/excalidraw/dist/prod/fonts/` are available for measurement
and rasterization. Unknown numeric `fontFamily` IDs in the scene fall
back to Excalifont metrics by default; `--strict-fonts` exits non-zero
with a list of offending families instead of substituting silently.

See [PLAN.md §4A.5](../PLAN.md), [FNT-009](../TASKS.md).

### CJK shard selection

Xiaolai (the CJK fallback) ships as 209 per-codepoint shards because no
single shard covers an arbitrary CJK string. The current
`FontkitTextMetricsProvider` picks the first shard containing the
requested codepoints; per-glyph fallback across multiple shards in a
single text run is documented as a TODO in
`src/core/text-metrics.mjs` (referenced from the FNT-005 completion
notes). For text that crosses shard boundaries, measurements may use a
single shard's metrics for the whole run. In practice this is bounded by
fontkit `layout()` and has not produced visible mis-wrapping on test
fixtures, but it is a known limitation rather than a guarantee.

See [PLAN.md §4A.4](../PLAN.md), [FNT-005](../TASKS.md) completion notes.

### Wrap parity

Per [PLAN.md §4A.2](../PLAN.md), the target tolerance is ±1 column per
line versus excalidraw.com. Our current regression gate
([T-006](../TASKS.md)) is **snapshot-based**: it locks the line-break
columns observed today and fails on drift, but it does not run the same
fixture against a live excalidraw.com browser oracle. The Playwright
infrastructure stood up for [FNT-005](../TASKS.md) is the natural place
to upgrade T-006 to a real column-level browser comparison; that upgrade
is a future step, not a current guarantee.

In practice, wrap behaviour matches the web app on all current fixtures
(`text-wrapped`, `wrap-400`, `mixed-script`).

### PNG text layout

`resvg`'s text engine does not re-wrap at PNG-rasterization time. This
is fine, because Excalidraw's SVG export already emits one `<text>`
element per wrapped line — the line-breaking decision was made by
Excalidraw's text engine, not the rasterizer. PNG output therefore
reflects the same wrapping as the SVG.

What PNG rasterization can drift on is **horizontal advance**: `resvg`
and Chromium use different shaping back-ends, so the visual gap between
glyphs may be sub-pixel different from a browser-rasterized SVG. For
Latin text in the bundled families this is not visible at typical export
sizes. For complex scripts (CJK, mixed-direction) the difference can be
larger; we do not currently characterize it. [PNG-004](../TASKS.md)
("PNG font fidelity") is the open task to lock SSIM ≥ 0.95 against a
headless-browser rasterization; until it lands, treat PNG text spacing
as approximate to the browser, exact to the SVG.

### Deterministic crypto polyfill

`deno_core`'s default runtime does not provide `crypto.getRandomValues`,
which Excalidraw's `nanoid` calls when generating element IDs at certain
points. The Rust shell installs a deterministic xorshift32 polyfill
([R-007](../TASKS.md) commit `da1112e`). On the parity gate's current
fixture set, none of these random bytes leak into the SVG output, so
Deno's real crypto and the Rust polyfill produce byte-identical results
coincidentally. If a future upstream change does cause `nanoid` output
to surface in the SVG, parity will break and the fix is to install the
same polyfill on the Deno side rather than special-case the diff.

This is a known load-bearing assumption: the parity gate does not prove
that the random bytes never matter, only that they do not matter for
today's fixtures.

## Known issues / open follow-ups

These are open against v1 and are not blockers for the documented
fidelity claims, but consumers should be aware of them.

- **Binary size: 87 MB on macOS arm64** (release build, LTO + strip).
  PHASE0.md §4 budgeted ~50 MB; PNG and embedded fonts pushed it past
  that. The 60 MB hard cap was knowingly exceeded; SZ-002 ([TASKS.md
  SZ-002](../TASKS.md)) is the regression gate that will lock the new
  budget once cross-compile lands.
- **C++ build-time dependency from PNG support.** WOFF2 decoding for
  the embedded `fontdb` uses `woofwoof` (Google's C++ woff2 + pure-Rust
  brotli) because every pure-Rust alternative tried in [PNG-001](../TASKS.md)
  failed on at least one of the 234 shipped shards. This makes
  cross-compile harder than a pure-Rust binary; the SZ-001 / REL-002
  matrix may need an alternative if `cc` cannot bridge to the target.
- **PNG fidelity gate ([PNG-004](../TASKS.md)) still `todo`.** The
  current PNG path produces visually correct output for the test
  fixtures, but the SSIM ≥ 0.95 gate against a headless-browser
  rasterization is not yet wired. Expect minor sub-pixel drift in glyph
  spacing relative to a Chromium-rendered SVG.
- **PNG fixture snapshots ([PNG-003](../TASKS.md)) still `todo`.** PNG
  output is exercised end-to-end but is not yet pixel-diffed against a
  committed golden the way SVG is.
- **V8 startup snapshot not used.** Cold start is ~220 ms median on
  `basic-shapes.excalidraw`; PHASE0.md set 80 ms in the bare F-002
  spike. The gap is shim install + Excalidraw module evaluation.
  PHASE0.md §"Residual open items" notes that a `deno_core` startup
  snapshot could cut this materially; deferred until cold start is the
  dominant complaint.
- **CJK per-glyph shard fallback** (above, "CJK shard selection"). The
  TODO sits in `src/core/text-metrics.mjs`; the FNT-005 baseline works
  around it by choosing a 12-character CJK string contained in a single
  shard.
- **T-006 column-level browser oracle.** The wrap-parity gate is
  snapshot-based today; upgrading it to a live excalidraw.com Playwright
  comparison is the natural follow-up to FNT-005 and would let us claim
  ±1 column per line directly rather than transitively through the
  snapshot.
- **macOS code-signing ([REL-003](../TASKS.md)) and crates.io publish
  ([REL-005](../TASKS.md)) pending.** Distribution-side, not
  fidelity-side, but listed here because they affect what users see when
  they install: unsigned macOS binaries trigger a Gatekeeper warning
  until REL-003 lands.

For the full task board including phases not relevant to fidelity, see
