# Fidelity reference

What `excalidraw-image` reproduces from the web app, what it deliberately
does not, and how each claim is tested.

## Goal

The SVG and PNG output, when rendered in a browser, should be visually
equivalent to what excalidraw.com produces, modulo the documented divergences
below.

"Equivalent" specifically means:

- Geometry, color, and stroke styles match at the SVG layer because the same
  upstream `exportToSvg` implementation runs inside the CLI.
- Text metrics, line breaks, and font selection match within measured
  tolerances.
- Editable `.excalidraw.svg` files round-trip: re-opening on excalidraw.com
  restores the original scene.

The CLI is not a pixel-exact reproduction of any specific browser engine.
Where measurable divergences exist, they are listed below with the mechanism
and the test that locks the current behaviour in place.

## What is exact

The export path is Excalidraw's own. The JS bundle ships
`@excalidraw/excalidraw`'s `exportToSvg`, the same function the web app calls.
All of the following are produced by upstream code:

- **Element placement, sizing, colors, opacity** — direct outputs of
  upstream's renderer.
- **Stroke styles, fill patterns** — generated by the bundled rough.js with
  the same seed semantics as the web app.
- **Text rendering algorithm** — Excalidraw's own line-break and metrics
  pipeline, with `FontkitTextMetricsProvider` replacing the browser's
  `CanvasRenderingContext2D.measureText`.
- **Font metric divergence <= 0.0005 px vs Chromium** for all eight supported
  families: Excalifont, Virgil, Nunito, Lilita One, Comic Shanns, Cascadia,
  Liberation Sans, and Xiaolai. Validated by
  `tests/js/browser-fidelity.test.mjs` against the committed Chromium
  `canvas.measureText` baseline in `tests/fixtures/browser-font-baseline.json`.
- **Editable `.excalidraw.svg` round-trips** — element IDs, `files` payloads,
  and font-family metadata are preserved through export and decode. Locked by
  `tests/js/embed-scene-roundtrip.test.mjs` and
  `tests/js/font-roundtrip.test.mjs`.
- **Byte-identical Deno/Rust output** — the dev-loop host (Deno) and the
  production host (`deno_core` in Rust) produce the same SVG bytes for every
  fixture in `tests/fixtures/`. Locked by
  `crates/excalidraw-image/tests/parity.rs`; drift is treated as a host leak,
  not as a tolerance to widen.

## Documented Divergences

### Emoji

Segoe UI Emoji is not bundled. Excalidraw's font fallback chain keeps
`Segoe UI Emoji` in the `font-family` attribute on emoji-bearing `<text>`
elements, but no `@font-face` declaration is emitted for it. Consequence:
emoji renders correctly only if the viewer's system has the font installed;
otherwise the glyphs fall back to a system default or appear as tofu. PNG
rasterization through `resvg` has the same limitation because `fontdb` is
loaded only with the bundled font set.

Locked by `tests/js/font-emoji-local-only.test.mjs`.

### Helvetica

Helvetica is not bundled. Text marked as Helvetica is measured against the
bundled Liberation Sans, which has near-equivalent metrics, but the emitted
SVG keeps `font-family="Helvetica, sans-serif, Segoe UI Emoji"` verbatim so
editable round-trips preserve the user's intent. In a browser with Helvetica
installed, the SVG will render in Helvetica; in one without it, the fallback
chain takes over.

Locked by `tests/js/font-helvetica-alias.test.mjs`.

### Embeddables

Embeddables such as iframes, YouTube, and video are not rendered.
`renderEmbeddables: false` is hard-coded in the JS export options. Scenes that
include embeddable elements export the underlying geometry, such as the
placeholder rectangle, but not the embedded content.

### OS-Local And User-Installed Fonts

Only the bundled Excalidraw WOFF2 families are available for measurement and
rasterization. Unknown numeric `fontFamily` IDs in the scene fall back to
Excalifont metrics by default; `--strict-fonts` exits non-zero with a list of
offending families instead of substituting silently.

Locked by `tests/js/font-strict-fonts.test.mjs`.

### CJK Shard Selection

Xiaolai, the CJK fallback, ships as many per-codepoint shards because no
single shard covers every CJK string. The current
`FontkitTextMetricsProvider` picks the first shard containing the requested
codepoints; per-glyph fallback across multiple shards in a single text run is
tracked in `src/core/text-metrics.mjs`. For text that crosses shard
boundaries, measurements may use a single shard's metrics for the whole run.
In practice this is bounded by fontkit `layout()` and has not produced visible
mis-wrapping on test fixtures, but it is a known limitation rather than a
guarantee.

### Wrap Parity

The target tolerance is +/-1 column per line versus excalidraw.com. The
current regression gate is snapshot-based: it locks the line-break behaviour
observed today and fails on drift, but it does not run the same fixture against
a live excalidraw.com browser oracle. The Playwright infrastructure used for
browser font measurements is the natural place to upgrade this to a real
column-level browser comparison.

In practice, wrap behaviour matches the web app on all current fixtures
(`text-wrapped`, `wrap-400`, `mixed-script`). Locked by
`tests/js/wrap-regression.test.mjs`.

### PNG Text Layout

`resvg`'s text engine does not re-wrap at PNG-rasterization time. This is
fine, because Excalidraw's SVG export already emits one `<text>` element per
wrapped line; the line-breaking decision was made by Excalidraw's text engine,
not the rasterizer. PNG output therefore reflects the same wrapping as the SVG.

What PNG rasterization can drift on is **horizontal advance**: `resvg` and
Chromium use different shaping back-ends, so the visual gap between glyphs may
be sub-pixel different from a browser-rasterized SVG. For Latin text in the
bundled families this is not visible at typical export sizes. For complex
scripts such as CJK or mixed-direction text, the difference can be larger and
is not yet characterized by an automated SSIM/browser raster gate. Treat PNG
text spacing as approximate to the browser and exact to the SVG.

### Deterministic Crypto Polyfill

`deno_core`'s default runtime does not provide `crypto.getRandomValues`, which
Excalidraw's `nanoid` calls when generating element IDs at certain points. The
Rust shell installs a deterministic xorshift32 polyfill. On the parity gate's
current fixture set, none of these random bytes leak into SVG output, so
Deno's real crypto and the Rust polyfill produce byte-identical results
coincidentally. If a future upstream change causes `nanoid` output to surface
in the SVG, parity will break and the fix is to install the same polyfill on
the Deno side.

This is a load-bearing assumption: the parity gate proves only that random
bytes do not matter for today's fixtures.

## What's Tested

| Area | Test or gate |
|---|---|
| Font inventory integrity | `tests/js/font-inventory.test.mjs` compares generated assets against `node_modules/@excalidraw/excalidraw/dist/prod/fonts/`. |
| Font subset round-trip | `tests/js/font-subset.test.mjs` verifies parseable WOFF2 subsets with metrics and cmap data. |
| Browser font metrics | `tests/js/browser-fidelity.test.mjs` compares fontkit widths to Chromium canvas widths. |
| Mixed-script fixture | `tests/fixtures/mixed-script.excalidraw` plus SVG golden/parity tests. |
| Wrap fixture | `tests/fixtures/wrap-400.excalidraw` plus `tests/js/wrap-regression.test.mjs`. |
| Helvetica aliasing | `tests/js/font-helvetica-alias.test.mjs`. |
| Strict font handling | `tests/js/font-strict-fonts.test.mjs`. |
| Emoji local-only policy | `tests/js/font-emoji-local-only.test.mjs`. |
| Editable SVG payload | `tests/js/embed-scene-roundtrip.test.mjs` and `tests/js/font-roundtrip.test.mjs`. |
| Deno vs Rust parity | `crates/excalidraw-image/tests/parity.rs`. |
| PNG end-to-end | `crates/excalidraw-image/tests/png.rs`. |
| SVG fixture snapshots | `tests/js/svg-goldens.test.mjs`; regenerate via `make goldens`. |

## Known Follow-Ups

These are not blockers for the documented fidelity claims, but consumers
should be aware of them.

- **Binary size:** release builds with full CJK support are large because V8,
  PNG support, and bundled fonts dominate. A future size pass could embed WOFF2
  and decode at runtime, or split CJK into an opt-in distribution package.
- **C++ build-time dependency from PNG support:** WOFF2 decoding for the
  embedded `fontdb` uses `woofwoof` because the pure-Rust alternatives tested
  failed on at least one shipped shard. This is the main reason Linux ARM64
  prebuilt binaries are deferred.
- **PNG browser/SSIM fidelity gate:** PNG output is exercised end-to-end but
  is not yet compared automatically to a headless-browser rasterization.
- **PNG binary goldens:** SVG goldens are committed; PNG output is covered by
  smoke tests rather than pixel-diffed committed snapshots.
- **V8 startup snapshot:** cold start is around 220 ms median on
  `basic-shapes.excalidraw`. A `deno_core` startup snapshot could reduce this,
  but has not been necessary so far.
- **CJK per-glyph shard fallback:** described above under CJK shard selection.
- **Column-level browser wrap oracle:** the current gate is snapshot-based;
  upgrading it to live browser comparison would tighten the wrap claim.
